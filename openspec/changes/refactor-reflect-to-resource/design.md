# Design: refactor-reflect-to-resource

## Decision Log

### D1: Resource subdir vs Area

**Options considered:**

| Location | Rationale for | Rationale against |
|---|---|---|
| `.wai/resources/reflections/` | PARA: Resources = reference material with no end date | Slightly non-standard (subdirs within resources usually hold agent-config, pipelines) |
| `.wai/areas/reflections/` | PARA: Areas = ongoing responsibilities | Areas hold project-like items, not file collections |
| `.wai/projects/<name>/reflections/` | Co-located with project artifacts | One dir per project; cross-project search is harder |

**Decision:** `.wai/resources/reflections/` — reflections are reference material
(like patterns/ and templates/) that spans projects and has no end date.
`config.rs` already has `resources_dir()` helper; adding a `REFLECTIONS_DIR`
constant follows the established pattern.

### D2: One file per run vs single accumulating file

**Options considered:**

| Approach | Pro | Con |
|---|---|---|
| One file per run (dated) | History preserved; easy to diff | Many small files; search returns multiple hits |
| Single file, overwritten | Simple; clean search results | No history |
| Single file, appended | History in one place | Non-structured; grows unbounded |

**Decision:** One file per run. Dated filenames (`YYYY-MM-DD-<project>.md`) sort
naturally. `wai search` returns multiple hits and the agent can read the most
recent. History lets future `wai reflect` runs include older reflections as
context (via `gather_reflect_context` which already reads from the resource dir
path if extended).

### D3: WAI:REFLECT:REF — generated by reflect vs by init

**Options considered:**

| Approach | Pro | Con |
|---|---|---|
| `wai reflect` writes the REF block | Only appears after first reflect run | `wai init` doesn't know about reflections |
| `wai init` always writes a REF placeholder | Consistent; agents see the search instruction from day one | Block appears before any reflections exist (pointing to empty dir) |

**Decision:** `wai init` writes a `WAI:REFLECT:REF` block unconditionally when
the managed block is injected. The block text is always useful ("search before
researching") even if no reflection files exist yet. `wai reflect` updates the
resource file but does NOT touch the REF block — that keeps the two concerns
separate.

### D4: Backwards compatibility for existing WAI:REFLECT blocks

Existing repos have `<!-- WAI:REFLECT:START -->` content in their CLAUDE.md.
On the next `wai reflect` run, the tool:
1. Detects the old block
2. Migrates its content to a dated resource file (`...-migrated.md`)
3. Replaces the block with the slim REF block
4. Prints a one-time migration notice

This migration runs only once (when the old block is present but no
`.wai/resources/reflections/` directory exists yet).

### D5: Where the "search before researching" instruction lives

The instruction needs to be in context on every session so agents actually follow
it. Two surfaces:

1. `WAI:REFLECT:REF` block in CLAUDE.md — always loaded (part of the file)
2. Managed block in `src/managed_block.rs` — written to new repos via `wai init`

Both should carry the instruction. The managed block version is the canonical
source; the REF block reinforces it specifically for the reflections resource.

## File Layout After Change

```
.wai/
└── resources/
    └── reflections/
        ├── 2026-02-28-friction-analysis.md   # from this session
        └── 2026-02-28-friction-analysis-migrated.md  # migrated from CLAUDE.md
```

CLAUDE.md slim block:
```markdown
<!-- WAI:REFLECT:REF:START -->
## Accumulated Project Patterns

Project-specific conventions, gotchas, and architecture notes live in
`.wai/resources/reflections/`. Run `wai search "<topic>"` to retrieve relevant
context before starting research or creating tickets.

> **Before research or ticket creation**: always run `wai search "<topic>"` to
> check for known patterns. Do not rediscover what is already documented.
<!-- WAI:REFLECT:REF:END -->
```

## Impact on `src/commands/reflect.rs`

| Function | Change |
|---|---|
| `run()` | Write to resource file instead of target file; detect+migrate old block |
| `gather_reflect_context()` | Also read `.wai/resources/reflections/` files as additional context tier |
| `inject_reflect_block()` | Rename/repurpose to `write_reflect_resource()` |
| `read_reflect_block()` | Read from resource dir (latest file) instead of target file |

## Impact on `src/managed_block.rs`

`wai_block_content()` gains a new paragraph in the "Starting a Session" or
"Before you start" area:

```
Before beginning research or creating a ticket, run `wai search "<topic>"` to
check for known patterns already accumulated in `.wai/resources/reflections/`.
```

Gated on `has_companions` (same as TDD disclaimer) — applies when beads or
openspec are present, i.e., when agents are doing implementation work.

## Impact on `src/managed_block.rs` — WAI:REFLECT:REF block

`inject_managed_block()` gains a secondary injection step: after injecting the
`WAI:START/END` block, check for and inject/update a `WAI:REFLECT:REF:START/END`
block with the slim reference text. This keeps the two blocks independent and
allows `wai init --refresh` to update both.
