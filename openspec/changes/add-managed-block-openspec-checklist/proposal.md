# Change: Add openspec steps to the managed block

## Why

The WAI:START block generated by `wai init` and `wai reflect` is the primary
guidance document AI assistants read at the start of every session. It already
includes a session-close checklist with beads steps (`bd close`, `bd sync`), but
has no mention of openspec when that plugin is active.

This creates a concrete, recurring failure: an AI completes implementation tasks
that correspond to openspec `tasks.md` checkboxes, closes the beads issues, and
commits — but never checks the boxes. The next session finds unchecked tasks and
can't tell whether the work is done or not. This is exactly what happened with
`add-why-agent-backend` and `doctor-fix-flag`.

There are two gaps to close:

1. **Session-close checklist** has no openspec step — adding
   `[ ] openspec tasks.md — mark completed tasks [x]` when openspec is detected
   closes the loop at the natural "done" moment.

2. **No cross-reference convention** — when beads and openspec are both active,
   there is no guidance on linking them. Defining the convention in the managed
   block means every AI session in every repo that uses both tools gets it
   automatically.

## What Changes

- **ADDED to session-close checklist** (when `has_openspec`): a step to mark
  completed openspec tasks `[x]` in `tasks.md`.
- **ADDED new section "Tracking Work Across Tools"** (when `has_beads &&
  has_openspec`): a short convention block explaining how to link beads tickets
  to openspec task IDs, and how to keep them in sync at close time.
- **No changes** to the starting-a-session section, quick reference, or any
  other part of the block.

## Impact

- Affected specs: new `managed-block` capability (no existing spec covers block
  content)
- Affected code: `src/managed_block.rs` — two conditional `push_str` additions
- No breaking changes; both additions are conditional on plugin detection
- Propagates to all repos on their next `wai init` or `wai reflect` run
